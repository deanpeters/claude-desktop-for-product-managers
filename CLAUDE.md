# CLAUDE.md

Operating contract for Claude (Desktop, Code, or any session) and human contributors working in this repository.

**First, check who you're talking to.** If this session was opened by a learner taking the course (they're asking about lessons, not about editing the repo), ignore the maintainer rules below: switch to `course-materials/COMPANION_INSTRUCTIONS.md` and be the course companion. The rest of this file is for people building the course.

## Read first

1. `README.md`
2. `CONSTITUTION.md`
3. `PLAN.md`
4. `docs/START_HERE.md`

## What this repo is

**Claude Desktop for Product Managers** — an interactive course teaching PMs to use Claude Desktop on real product work. Learners point a Cowork session at `course-materials/` and are guided lesson by lesson. This is a teaching product, not a codebase.

## Non-negotiables

1. **Original work.** Do not import, paraphrase, or lightly rewrite content from third-party courses, repos, or websites. Structure inspiration is fine; copied copy is not.
2. **No real company names** that trace to a client, prospect, competitor, or vendor engagement — anywhere, ever, including as prohibited examples. Fictional companies only, and they must not echo a real one. If in doubt, stop and ask Dean.
3. **No fake magic.** Never claim a Claude Desktop feature is wired up when it isn't. If a lesson needs setup, say so plainly.
4. **No hype copy.** Lessons teach a workflow with visible judgment, or they don't ship.
5. **No secrets** — no tokens, credentials, or real client data in any file.

## Working style

- Draft first: produce the most complete version before asking for sign-off.
- Optimize for a reader who is not a daily terminal user. GUI-first framing; the terminal is never required.
- Keep lesson structure identical across modules — see `docs/course-architecture.md` for the six-field lesson shape.
- Markdown-first, `.md` extension for every learner-facing file.
- Small named files over monoliths. Keep the front door (README) honest about current state.
- Voice: direct, practical, warm, concise. Outcomes over outputs. No corporate beige, no buzzword fog.

## Verification

- Check internal links and file paths when editing Markdown.
- Walk one learner path end-to-end after structural changes: can a PM open the files, follow the prompts, and judge the result?
